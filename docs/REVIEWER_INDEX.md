# Reviewer Index

## Purpose

This index gives reviewers a clean order for reading the consequence-boundary infrastructure category surface.

## Recommended review order

1. `README.md`

   Establishes the category:

   ```text
   consequence-boundary infrastructure
   ```

2. `docs/EXECUTIVE_ONE_PAGER.md`

   Gives the shortest enterprise explanation.

3. `docs/BUYER_BRIEF.md`

   Explains the buyer problem and commercial positioning.

4. `docs/ENTERPRISE_PROBLEM_FIELDS.md`

   Maps the five enterprise problem fields:

   ```text
   privacy / data movement
   AI output to action
   aerospace / engineering stage admission
   enterprise access / privileged operations
   AI memory / identity / agent continuity
   ```

5. `docs/CORRIDOR_PORTFOLIO.md`

   Shows how each problem field becomes a bounded corridor.

6. `docs/PILOT_SELECTION_GUIDE.md`

   Explains why Privacy Action Custody is the recommended first pilot.

7. `docs/FIRST_CORRIDOR_PRIVACY_ACTION_CUSTODY.md`

   Details the first commercial corridor.

8. `docs/OUTCOME_POSTURE_MAP.md`

   Defines why outcomes must control physical posture.

9. `docs/RECEIPT_REPLAY_REQUIREMENTS.md`

   Defines receipt/replay evidence requirements.

10. `docs/UNDENIABLE_STANDARD.md`

    Defines what makes a corridor inspectable and hard to misclassify.

11. `docs/ENTERPRISE_DISCOVERY_QUESTIONS.md`

    Converts the category into client discovery.

12. `docs/PUBLIC_PROTECTED_BOUNDARY.md`

    Clarifies what is public and what remains protected.

## Reviewer takeaway

```text
This repository does not claim to replace enterprise workflows.

It defines the missing boundary where high-risk workflow output becomes consequence.
```

## Clean conclusion

```text
The category is unified.
The corridors remain bounded.
The protected implementation stays private.
```
