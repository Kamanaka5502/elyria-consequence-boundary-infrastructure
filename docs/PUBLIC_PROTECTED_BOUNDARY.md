# Public / Protected Boundary

## Purpose

This repository is the public category surface for consequence-boundary infrastructure.

It must explain the category clearly without exposing protected implementation, client packets, private kernels, production adapters, law bundles, scoring laws, or commercial corridor internals.

## Public surface

The public repository may expose:

```text
category language
enterprise problem fields
corridor portfolio
buyer brief
pilot selection guide
undeniable standard
outcome posture map
receipt and replay requirements
safe first-corridor framing
public discovery questions
```

## Protected material

The public repository must not expose:

```text
private kernels
client-specific law bundles
client packet internals
production adapters
runtime substrate
scoring laws
commercial implementation logic
system-specific integration maps
key management details
access-control internals
immutable storage implementation
customer-specific evidence mappings
```

## Why this boundary matters

The public category must be strong enough for buyers, reviewers, and partners to understand the market position.

The protected layer must remain controlled enough to preserve commercial value and prevent derivative implementation.

## Correct public posture

Use:

```text
public category surface
bounded proof surface
controlled evaluation corridor
receipt/replay direction
consequence-boundary infrastructure
```

Do not use:

```text
open-source runtime
production deployment package
complete implementation
client law bundle
full protected substrate
commercial adapter source
```

## Repository rule

```text
This repository explains the category.
It does not transfer the machinery.
```

## Clean line

```text
Public gets the shape of the boundary.
Protected keeps the engine that enforces it.
```
